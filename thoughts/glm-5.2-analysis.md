# GLM-5.2 深度分析

> 2026-06-18，数据来源 HuggingFace API 源码验证

---

## 为什么重新看 GLM-5.2

之前（2026-06-16 发布当天）写过初步评估，当时参数规模未公开，结论是「等一个月看」。

一周后我直接从 HuggingFace 拉了完整配置和 safetensors 数据，把架构和显存算清楚了。

## 从 HuggingFace 拉到的配置

官方仓库：`zai-org/GLM-5.2-FP8`

| 参数 | 值 |
|---|---|
| 架构 | `glm_moe_dsa`（DS-Attention MoE） |
| 总参数（safetensors 实测） | **~751B FP8 + ~2.1B BF16** |
| 层数 | 78（前 3 层稠密，75 层 MoE） |
| Hidden size | 6144 |
| Attention heads | 64，KV heads 64 |
| 路由专家 | **256** |
| 共享专家 | 1 |
| 每 token 激活 | 8 个 expert |
| MoE intermediate size | 2048（每 expert） |
| 稠密层 intermediate | 12288 |
| 上下文长度 | **1,048,576（1M）** |
| QKV 机制 | MLA（q_lora_rank=2048, kv_lora_rank=512） |
| 词表 | 154,880 |
| 许可证 | MIT ✅ |
| 下载量 | FP8 版 3,689，原始版 666 |

### 参数推算

75 层 MoE，每层：
- MLA 注意力：~160M
- 256 个 expert FFN：256 × (6144×2048×3) ≈ 9.68B
- 1 个共享 expert：~38M
- 总计每 MoE 层 ≈ 9.9B

75 层 × 9.9B = 742.5B

3 层稠密层：
- 注意力 + FFN（12288）：~385M × 3 ≈ 1.16B

Embedding + LM Head（如果绑定）：~952M

**总计 ≈ 745B，和 safetensors 实测 ~751B 吻合。**

## 显存需求

| 量化 | 所需显存 | 2×A100 160GB | 双 B200 384GB | 8×A100 640GB |
|---|---|---|---|---|
| FP8（官方）| ~751 GB | ❌ | ❌ | ✅ |
| Q4（4-bit GGUF）| ~376 GB | ❌ | ✅ 刚好 | ✅ |
| Q3（3-bit）| ~282 GB | ❌ | ✅ | ✅ |
| Q2（2-bit）| ~188 GB | ❌ | ✅ | ✅ |

**结论：方案 B（双 A80 160GB）完全装不下。双 B200 384GB 在 Q4 下刚好能装，但余量极少。**

## 值不值得

### 优点
- 1M 上下文：IndexShare 技术，2.9× FLOPs 减少
- SWE-bench Pro 62.1（编程国产第一）
- Terminal-Bench 81.0（Agent 场景最强）
- MIT 许可证

### 缺点
- **745B 参数**——太大了，部署门槛极高
- 刚发布一周，社区验证极少（下载量 3689 vs Qwen3.5-122B 的 76 万）
- GLM-5/5.1 前代口碑差：服务稳定性问题、乱码、token 消耗黑洞
- 即使双 B200 Q4 勉强装下，也没有余量做并发

### 和 Qwen3.5-122B 对比

| | GLM-5.2 | Qwen3.5-122B-A10B |
|---|---|---|
| 总参数 | 745B | 122B |
| 激活参数 | ~2T FLOPs/词 (8/256) | 10B |
| 部署门槛 | 8×A100 / 双B200 Q4 | B200 FP8 / 双A80 Q4 |
| 生态 | 1 周 | 6 个月 |
| 中文质检 | 无权威测评 | SuperCLUE 已验证 |

## 最终判断

GLM-5.2 的定位是 **超大规模集群模型**（8×H100 起步），和我们的双 A80 / 单 B200 路线不是一个赛道。硬件规模匹配不上模型规模，不推荐。如果未来出了蒸馏/剪枝版的小模型（比如 GLM-5.2-52B），可以重新评估。
